## 1. 인수 기준 개요 (Acceptance Criteria Overview)

## 2. 매수 주문 시나리오 (Buy Order Scenarios)

### AC-001: 정상 매수 주문 플로우

**Feature**: 매수 주문 실행

**Scenario**: 사용자가 충분한 KRW 잔액으로 매수 주문을 성공적으로 완료

```gherkin
Given 사용자가 "KRW-BTC" 마켓을 선택하고
And 사용자의 KRW 잔액이 50,000원 이상이고
And 사용자가 매수 금액으로 10,000원을 입력하고
When 사용자가 "매수 주문" 버튼을 클릭하면
Then 시스템은 승인 다이얼로그를 표시한다
And 승인 다이얼로그에는 예상 체결가가 표시된다
And 승인 다이얼로그에는 예상 수수료가 표시된다
And 승인 다이얼로그에는 "승인"과 "취소" 버튼이 있다

When 사용자가 "승인" 버튼을 클릭하면
Then 시스템은 Upbit API /v1/orders를 호출한다
And 시스템은 주문 UUID를 반환한다
And UI에 "주문 완료" 메시지가 표시된다
And 잔액이 자동으로 새로고침된다
And KRW 잔액이 10,000원 + 수수료만큼 감소한다
```

**검증 방법**:
- [ ] UI 자동화 테스트 (Playwright)
- [ ] API 호출 Mock 검증
- [ ] 잔액 변화 확인

### AC-002: 잔액 부족으로 매수 주문 실패

**Feature**: 매수 주문 검증

**Scenario**: 사용자의 KRW 잔액이 부족하여 매수 주문이 차단됨

```gherkin

And 사용자의 KRW 잔액이 3,000원이고
And 사용자가 매수 금액으로 10,000원을 입력하고
When 사용자가 "매수 주문" 버튼을 클릭하면
Then 시스템은 승인 다이얼로그를 표시하지 않는다
And UI에 "잔액이 부족합니다" 에러 메시지가 표시된다
And 시스템은 Upbit API를 호출하지 않는다
```

**검증 방법**:
- [ ] 에러 메시지 한국어 확인
- [ ] API 호출 발생하지 않음 확인

## 3. 매도 주문 시나리오 (Sell Order Scenarios)

### AC-010: 정상 매도 주문 플로우

**Feature**: 매도 주문 실행

**Scenario**: 사용자가 충분한 코인 보유량으로 매도 주문을 성공적으로 완료

```gherkin
Given 사용자가 "KRW-BTC" 마켓을 선택하고
And 사용자가 0.01 BTC를 보유하고 있고
And 사용자가 매도 수량으로 0.005 BTC를 입력하고
When 사용자가 "매도 주문" 버튼을 클릭하면
Then 시스템은 승인 다이얼로그를 표시한다
And 승인 다이얼로그에는 예상 체결가가 표시된다
And 승인 다이얼로그에는 예상 수령 금액이 표시된다

When 사용자가 "승인" 버튼을 클릭하면
Then 시스템은 Upbit API /v1/orders를 호출한다 (side=ask)
And UI에 "주문 완료" 메시지가 표시된다
And BTC 보유량이 0.005만큼 감소한다
And KRW 잔액이 증가한다
```

**검증 방법**:
- [ ] 코인 보유량 감소 확인
- [ ] KRW 잔액 증가 확인
- [ ] API 호출 파라미터 확인 (side=ask)
